# Secret rotation and audit logging

Companion to the [Production Hardening Guide](production-hardening.md) and [Secrets Management](secrets-management.md). Covers parent issue [#128](https://github.com/JoshLuedeman/cosmosdb-to-sql-migration-tool/issues/128) sub-task [#203](https://github.com/JoshLuedeman/cosmosdb-to-sql-migration-tool/issues/203).

This guide assumes you have already adopted the managed-identity, RBAC, and network postures from #199–#202. After those, the **CosmosToSqlAssessment** tool itself has no static secrets at runtime — the only rotatable surfaces left are produced by the SQL Database Project the tool *generates* and by your chosen CI/CD wiring.

---

## Rotation surfaces inventory

| Surface | Where it lives | Section |
|---|---|---|
| SQL deploy credentials (`publish.xml`, `$(SqlPassword)`, `Deploy-SqlProject.ps1 -Password`) | Generated by `Services/SqlProjectIntegrationService.cs` for the operator's CI/CD pipeline | [SQL deploy passwords](#sql-deploy-passwords) |
| Service-principal client secret (Pattern A SP, no OIDC) | Azure DevOps / GitHub Actions secret store + Key Vault | [Service-principal client secret](#service-principal-client-secret) |
| Federated identity credential (AKS Workload Identity) | Managed identity trust configuration | [Federated credentials are not secrets](#federated-credentials-are-not-secrets) |
| Cosmos DB account primary/secondary keys (residual) | Cosmos account, even with `disableLocalAuth=true` | [Cosmos account key residual exposure](#cosmos-account-key-residual-exposure) |

The runtime tool's managed-identity token is rotated automatically by Microsoft Entra (typically every 1–24 hours depending on `tokenCredentialOptions`) — no operator action required.

## The preferred path: eliminate the rotation problem

The strongest "rotation procedure" is the one you never have to run. Each rotation surface above has a Entra-only alternative that removes the static credential entirely:

| If you are doing this | Switch to this | Result |
|---|---|---|
| SQL deploy with `SqlPassword` in the pipeline | sqlpackage with `/Authentication:ActiveDirectoryServicePrincipal` (Azure DevOps `SqlAzureDacpacDeployment@1` task `AuthenticationType: servicePrincipal`) targeting a service principal added to an Entra group that is mapped to `db_owner` / `db_schema_deployer` in the database | No SQL password to rotate. SP secret rotation handled by Key Vault |
| Azure DevOps service connection with SP secret | Azure DevOps **workload identity federation** for the service connection | No SP secret; tokens are exchanged per-run, expire immediately after the pipeline finishes |
| GitHub Actions with `creds: ${{ secrets.AZURE_CREDENTIALS }}` | `azure/login@v2` with OIDC federated credential on a user-assigned managed identity | No SP secret in repo or org secrets |
| Container Apps / App Service `KeyVaultReference` to a SQL password | Use Entra-only SQL connection string; principal = the host's managed identity already added to Entra group + DB | Removes the Key Vault secret as a rotation surface |

If you have adopted all four switches above, your rotation runbook is one line long: **"There are no secrets to rotate."** Skip to [audit logging](#audit-logging-recipe).

## Rotation fallback procedures

Tenants that cannot fully eliminate static credentials (corporate-policy reasons, on-prem CI/CD agents, vendor lock-in, etc.) should adopt Azure Key Vault native rotation for everything they cannot migrate off.

### Pattern: Key Vault rotation policy + Event Grid

Key Vault supports a native rotation policy that emits a `Microsoft.KeyVault.SecretNearExpiry` event ~30 days before a secret's `validityPeriod` ends. Wire that event to an Azure Function (or Logic App) that generates a new secret value, applies it to the target resource, and writes the new version back to Key Vault.

The full Function-rotation scaffold is documented by Microsoft:

> **[Tutorial: Automate the rotation of a secret for resources that use one set of authentication credentials](https://learn.microsoft.com/en-us/azure/key-vault/secrets/tutorial-rotation)**

Below is the "what your rotation Function actually needs to do" per secret kind — the parts the Microsoft tutorial does not specialize.

#### SQL deploy passwords

If you are rotating a **SQL logical server admin** password:

```powershell
# Inside your rotation Function (running as a managed identity with Az.Sql + Key Vault access)
$newPassword = New-SecurePassword -Length 32  # use a real generator
Set-AzSqlServer -ResourceGroupName $rg -ServerName $server -SqlAdministratorPassword $newPassword
Set-AzKeyVaultSecret -VaultName $vault -Name 'sql-admin-password' -SecretValue $newPassword
```

If you are rotating a **contained database user / non-admin SQL login** password (the more common case for `sqlpackage` deploy accounts):

```sql
-- Run inside the target database
ALTER LOGIN [svc-sqlpackage] WITH PASSWORD = '<new-password>' OLD_PASSWORD = '<old-password>';
```

Either way, do **not** use `Set-AzSqlServerActiveDirectoryAdministrator` here — that command sets/replaces the Entra admin and does nothing to a SQL Authentication password. Two different mechanisms; mixing them up has corrupted production deploys.

Recommended cadence: every 90 days, or after any suspected exposure. Always overlap (write the new version, then update consumers, then delete the old version after one full deployment cycle has verified the new one works).

#### Service-principal client secret

```bash
# Add a NEW credential without revoking the existing one
az ad app credential reset \
  --id <app-object-id> \
  --append \
  --years 1 \
  --display-name "rotation-$(date +%Y%m%d)"

# Store the returned password in Key Vault as a new secret version
az keyvault secret set --vault-name $vault --name sp-client-secret --value "<password-from-above>"

# Wait at least 24h for downstream consumers to pick up the new secret, then:
az ad app credential delete --id <app-object-id> --key-id <old-key-id>
```

The `--append` flag is mandatory. Omitting it overwrites **all** existing credentials on the application — a one-keystroke way to invalidate every running pipeline at once. The Microsoft docs default to `--append=False`, which is why this trip-wire exists.

Recommended cadence: every 90 days. Microsoft's "managed identity instead" recommendation applies — every SP secret you rotate is a SP secret you could have eliminated.

### Federated credentials are not secrets

Federated identity credentials (AKS Workload Identity, GitHub OIDC, Azure DevOps OIDC) do not have an expiry to rotate. They are a **trust relationship** between an external token issuer and a Microsoft Entra application or user-assigned managed identity. Three legitimate operations against a federated credential:

- **Update** — when the K8s ServiceAccount name, namespace, GitHub repo, or branch changes:
  ```bash
  az identity federated-credential update \
    --name <fic-name> --identity-name <uami> --resource-group <rg> \
    --subject "system:serviceaccount:<new-ns>:<new-sa>"
  ```
- **Delete and recreate** — when the trust must be revoked (e.g., the cluster was decommissioned).
- **Replace the underlying UAMI** — when the UAMI itself is compromised. Create a new UAMI, reassign all four required RBAC roles (see [Production Hardening](production-hardening.md)), update the K8s ServiceAccount `azure.workload.identity/client-id` annotation, then delete the old UAMI.

There is no "rotate every 90 days" procedure. Replace on change-of-trust or compromise; otherwise leave alone. Audit failures land in `AADServicePrincipalSignInLogs` (see [detection #5 below](#5-workload-identity-federated-credential-failures)).

### Cosmos account key residual exposure

Even with `disableLocalAuth=true` set on the account, Cosmos DB primary/secondary keys still **exist** in the account resource and can be retrieved by any principal holding `Microsoft.DocumentDB/databaseAccounts/listKeys/action`. The custom ARM Reader role in [`security/rbac/cosmos-assessment-arm-reader.json`](security/rbac/cosmos-assessment-arm-reader.json) deliberately omits this action, but other principals (subscription Owner, Contributor, the built-in Cosmos DB Operator) still have it.

Defense in depth:

1. **Keep `disableLocalAuth=true`.** This makes key-based auth fail at the data plane even if a key leaks.
2. **Audit `listKeys` / `regenerateKey`** as a high-priority alert (see [detection #1 below](#1-arm-listkeys--regeneratekey-on-cosmos-accounts)).
3. **Rotate keys** with `az cosmosdb keys regenerate --key-kind primary` (and again with `--key-kind secondary` after a few hours) any time you suspect exposure, *and* before any planned emergency re-enable of local auth.
4. **Inventory who holds ARM roles** that include `listKeys/action` quarterly. The smaller this set, the less rotation pressure.

## Audit logging recipe

Five diagnostic-setting destinations, one Log Analytics workspace, five starter detections. Send everything to the same workspace that the assessment tool reads from — the tool's managed identity already has `Log Analytics Reader` per #199/#202, so no additional grants are required.

> **About resource-specific tables:** the queries below show both the dedicated table (e.g. `CDBDataPlaneRequests`) and the legacy `AzureDiagnostics` fallback. The dedicated table is preferred (cheaper, typed columns) but **requires** the diagnostic setting to be created with `LogAnalyticsDestinationType=Dedicated`. Without that flag, the dedicated table will be empty and the fallback is what you must query. See [Azure Monitor resource logs](https://learn.microsoft.com/en-us/azure/azure-monitor/essentials/resource-logs) for the export format.

### Enable diagnostic settings

| Service | Categories to enable | Destination | Notes |
|---|---|---|---|
| Cosmos DB | `DataPlaneRequests`, `ControlPlaneRequests` | Log Analytics (Dedicated) | `PartitionKeyStatistics` / `PartitionKeyRUConsumption` are **operational** telemetry, not audit; enable only if you also want capacity-planning data and accept the cost + the partition-key values they may surface |
| Key Vault | `AuditEvent`, `AzurePolicyEvaluationDetails` | Log Analytics (Dedicated) | 90-day workspace retention minimum; for SOC 2 / FedRAMP, also tee to immutable Storage with a 7-year retention rule |
| Azure SQL Database | `SQLSecurityAuditEvents` (via [SQL Audit](https://learn.microsoft.com/en-us/azure/azure-sql/database/auditing-overview)) | Log Analytics or Storage | Configure with `Set-AzSqlDatabaseAudit` |
| Microsoft Entra | `SignInLogs`, `AuditLogs`, `NonInteractiveUserSignInLogs`, `ServicePrincipalSignInLogs`, `ManagedIdentitySignInLogs` | Log Analytics | Requires Entra ID P1 or P2 |
| Azure subscription activity | All categories | Log Analytics | Implicitly captures `Microsoft.DocumentDB/databaseAccounts/listKeys/action` and `regenerateKey/action` |

Example: enable Cosmos diagnostic with the Dedicated table type:

```bash
az monitor diagnostic-settings create \
  --name "cosmos-to-law" \
  --resource "/subscriptions/<sub>/resourceGroups/<rg>/providers/Microsoft.DocumentDB/databaseAccounts/<cosmos>" \
  --workspace "/subscriptions/<sub>/resourceGroups/<rg>/providers/Microsoft.OperationalInsights/workspaces/<law>" \
  --export-to-resource-specific true \
  --logs '[{"category":"DataPlaneRequests","enabled":true},{"category":"ControlPlaneRequests","enabled":true}]'
```

The `--export-to-resource-specific true` flag is the CLI form of `LogAnalyticsDestinationType=Dedicated`.

### Enable Microsoft Defender for Cloud

Both plans are subscription-scoped and bill per-resource:

- **[Microsoft Defender for Key Vault](https://learn.microsoft.com/en-us/azure/defender-for-cloud/defender-for-key-vault-introduction)** — detects anomalous secret access, suspicious key-listing, geographic anomalies.
- **[Microsoft Defender for Databases](https://learn.microsoft.com/en-us/azure/defender-for-cloud/defender-for-databases-introduction)** — the 2024 umbrella plan covering Azure SQL **and** Cosmos DB; surfaces SQL injection attempts, data exfiltration patterns, anomalous query workloads.

Enable both via portal *or*:

```bash
az security pricing create --name KeyVaults --tier Standard
az security pricing create --name CosmosDbs --tier Standard
az security pricing create --name SqlServers --tier Standard
```

Defender alerts land in `SecurityAlert` in your Log Analytics workspace and the Defender for Cloud Alerts blade.

### KQL detection library

Five queries covering the residual abuse paths the assessment tool's surface area exposes. Each maps to a scheduled-alert rule (the suggested threshold/window is indicative — tune to your environment).

#### 1. ARM `listKeys` / `regenerateKey` on Cosmos accounts

After `disableLocalAuth=true`, these operations should be rare and always investigated. They are **ARM** operations, so they land in `AzureActivity`, not in `CDBControlPlaneRequests`.

```kusto
AzureActivity
| where TimeGenerated > ago(1h)
| where OperationNameValue in (
    "MICROSOFT.DOCUMENTDB/DATABASEACCOUNTS/LISTKEYS/ACTION",
    "MICROSOFT.DOCUMENTDB/DATABASEACCOUNTS/REGENERATEKEY/ACTION"
)
| where ActivityStatusValue == "Success"
| project TimeGenerated, OperationNameValue, Caller, CallerIpAddress, _ResourceId, Properties
```

**Alert:** trigger on any result. Frequency 5 min, window 15 min, severity High.

#### 2. Cosmos data-plane RBAC writes

These are role-definition / role-assignment changes inside the Cosmos account — distinct from ARM activity. Land in `CDBControlPlaneRequests` (dedicated) or `AzureDiagnostics` (legacy).

```kusto
// Dedicated table (LogAnalyticsDestinationType=Dedicated)
CDBControlPlaneRequests
| where TimeGenerated > ago(1h)
| where operationName_s has_any (
    "sqlRoleDefinitions",
    "sqlRoleAssignments"
)
| where httpStatusCode_s startswith "2"
| project TimeGenerated, operationName_s, callerIdentity_s, requestResourceType_s, _ResourceId

// Legacy fallback
AzureDiagnostics
| where Category == "ControlPlaneRequests"
| where ResourceProvider == "MICROSOFT.DOCUMENTDB"
| where operationName_s has_any ("sqlRoleDefinitions", "sqlRoleAssignments")
| where statusCode_s startswith "2"
```

**Alert:** trigger on any result. Frequency 15 min, window 60 min, severity High.

#### 3. Key Vault SecretGet by unexpected principal

```kusto
// Replace EXPECTED_PRINCIPALS with the object IDs of the assessment-tool MI + any
// human break-glass identities you accept.
let EXPECTED_PRINCIPALS = dynamic([
    "00000000-0000-0000-0000-000000000001",   // tool UAMI
    "00000000-0000-0000-0000-000000000002"    // sec-ops break-glass
]);

// Dedicated table
AzureKeyVaultAuditEvents
| where TimeGenerated > ago(1h)
| where OperationName == "SecretGet" and ResultType == "Success"
| where identity_claim_oid_g !in (EXPECTED_PRINCIPALS)
| summarize Count = count(), Secrets = make_set(id_s) by identity_claim_oid_g, CallerIPAddress
| where Count > 0

// Legacy fallback
AzureDiagnostics
| where ResourceType == "VAULTS" and Category == "AuditEvent"
| where OperationName == "SecretGet" and ResultType == "Success"
| where identity_claim_oid_g !in (EXPECTED_PRINCIPALS)
```

**Alert:** any result. Frequency 5 min, window 15 min, severity Medium.

Time-of-day filters are deliberately omitted — the assessment tool may run ad hoc at any hour. If you need to lower false-positive rate, raise the threshold on burst access volume or filter by specific high-value secret names rather than by time.

#### 4. SQL deploy logins by principal outside the deployer group

Assumes Azure SQL Audit is enabled and `sg-sql-deployers` is the Entra group that backs the `db_schema_deployer` role.

```kusto
// Legacy AzureDiagnostics path (most common deployment)
AzureDiagnostics
| where TimeGenerated > ago(1h)
| where Category == "SQLSecurityAuditEvents"
| where action_name_s in ("BATCH COMPLETED", "DATABASE AUTHENTICATION SUCCEEDED")
| where database_name_s == "<your-database>"
| where server_principal_name_s !contains "sg-sql-deployers"
| where server_principal_name_s !in ("<service-account-1>", "<service-account-2>")
| project TimeGenerated, server_principal_name_s, application_name_s, client_ip_s, statement_s

// Dedicated table (if SQL audit was configured with Dedicated destination)
// SQLSecurityAuditEvents
// | where ...
```

**Alert:** any result not in the allowlist. Frequency 5 min, window 15 min, severity Medium.

#### 5. Workload-identity federated-credential failures

Often the first symptom of a tampered K8s ServiceAccount annotation, a deleted federated credential, or an issuer/audience drift after an AKS cluster upgrade.

```kusto
AADServicePrincipalSignInLogs
| where TimeGenerated > ago(1h)
| where isnotempty(FederatedCredentialId)
| where ResultType != 0
| summarize FailCount = count(), Reasons = make_set(ResultDescription)
    by ServicePrincipalId, ServicePrincipalName, FederatedCredentialId
| where FailCount > 5
```

**Alert:** more than 5 failures in 15 minutes. Severity Medium. Tune `FailCount` to your environment baseline.

## Putting it together: a quarterly rotation runbook

For tenants that adopted the [preferred path](#the-preferred-path-eliminate-the-rotation-problem), the runbook is short:

1. **Audit who can still call `listKeys` / `regenerateKey`** on Cosmos accounts. Trim the role-assignment list if it has grown.
2. **Review SP secrets across all subscriptions** that touch this tool. Any non-zero count is a candidate for migration to OIDC federated credentials.
3. **Confirm Defender for Cloud plans are still enabled** (someone disables them every six months "to save money").
4. **Replay the five KQL queries above** over the last 90 days. Investigate any non-empty result.
5. **Verify workspace retention** is still at or above your compliance baseline.

For tenants on the fallback path, add:

6. **Run the rotation Function manually** against each managed secret to verify the path works end-to-end. Better to find a broken rotation pipeline now than at 03:00 on the day a secret expires.
7. **Confirm each `KeyVaultReference` consumer** picked up the latest secret version (Container Apps revisions, App Service config refresh, AKS `secretProviderClass` sync).
